<h2>2. 채택한 개발 기술과 브랜치 전략</h2>
Github Flow
<br>

- 브랜치 <br>
1 . 각 개인 작업용 브랜치 <br>
2 . 개발용 Dev 브랜치<br>
3 . 배포용 EC2(Prd) 브랜치 <br><br>
- Github Convention <br> 1.  초기 세팅을 완료한 Main 브랜치로 시작 <br> 2. 각 개인 개발자가 Main 브랜치를 Checkout, 개발 시작 <br> 3. 두명씩 두개의 조로 나뉘어 개발중인 기능 테스트가 필요 할 시 각 조의 브랜치에서 테스트  <br> 4. 매일 PM 5:50에 Pull Request <br> 5. 모든 PR요청 완료시 팀원들은 사용중이던 Remote 브랜치를 제거 후 팀장이  점검/테스트 후 Checkout 해서 작업 진행

<h2>4. 페이지별 기능</h2>



- login(Spring Security, JWT)<br>
일반로그인 : 홈페이지 내에서 회원가입 한 사용자 로그인<br>
자동로그인 : 자동로그인 체크 후 로그인 시 1주일 간 자동로그인<br>
아이디저장 : 최근 로그인 한 사용자 아이디를 저장<br>
소셜로그인 : 카카오, 네이버, 구글

- join<br>
회원가입 시 필요한 정보를 입력하여 저장<br>
수의사인증 : 이름, 수의사번호로 수의사인증 가능

- findId<br>
이름, 연락처를 입력하면 해당하는 아이디를 보여줌<br>
아이디 찾기 완료 후 비밀번호 찾기 페이지로 이동 가능

- findPw<br>
이름, 아이디를 입력하고 임시비밀번호를 발급하면 아이디(메일주소)로 임시 비밀번호 발급

- userDetail<br>
현재 로그인한 사용자의 정보를 보여줌<br>
회원탈퇴 : 회원탈퇴 기능. 회원탈퇴 시 글, 댓글, 좋아요, 스크랩, 신고 제거

- userModify<br>
현재 로그인한 사용자의 정보를 수정하는 페이지<br>
모든 항목은 기존 비밀번호를 입력해야 수정 가능<br>
수의사인증 : 이름, 수의사번호로 수의사인증 가능

- 구현석 (개인) : <br>
페이지 별 구성, 색상 등의 디자인을 같은 사이트라고 인식할 수 있게끔 템플릿을 구상했다.<br>
버튼의 위치를 사용자의 입장에서 편의성을 고려하여 배치하였고 색상이 템플릿의 일부로 느끼게끔 하면서 기능이 맞춘 hover 색상을 선정하였다.<br>
게시물의 댓글의 등록, 삭제를 구현하고 가시성을 위해 댓글 접기 기능을 활용하였다.<br>
게시판 이용시 편의성을 위해 우측에 플로팅 메뉴를 추가하였고, 사용자의 스크롤 위치에 따라 같이 움직이도록 노력하였다.<br>
사이트의 전체적인 분위기를 해치지 않는 이미지를 선정하려고 노력하였다.<br>
부트스트랩을 활용하여 전체적인 틀을 구상하고 세부적인 내용은 자체적으로 작성한 CSS로 해결하려고 노력했습니다.<br>

- 김주희 (개인) : <br>
에러 처리 부분의 가시성을 향상시키기 위해 노력<br>- error를 직접 발생시켜보니 CustomRestAdvice 쪽의 에러는 알아보기 힘든 부분이 있어 에러를 보고 위치를 추측할 수 있도록 전반적으로 모든 에러 코드를 수정하였습니다.<br><br>
페이지 및 키워드를 유지 구분<br>- read 페이지에 접속하는 경로가 여러가지라 read페이지에 접속한 후 목록버튼을 통해 되돌아갈때 어떤 방식으로 페이지 및 키워드를 유지 시킬지 기준을 세웠고 그 기준을 통해 구분하도록 하였습니다.<br><br>
필터 연결<br>- 사용자가 적용시킬 수 있는 필터와 페이지에 접속할때 기본 적용되어 있는 필터들을 자연스럽게 연결하기위해 노력하였습니다.<br><br>
같은 template에서의 출력 구분<br>- 같은 template이라도 조건에 따라 다르게 보여질 수 있도록 구분하였습니다.<br>

- 팀장 역할<br><br> -  팀 전체의 역량과 개발 일정을 생각해 팀원들의 아이디어를 모으며 수용하되 과열되지 않도록 조율했습니다<br><br> - 팀원들의 작업 방식을 통일 시키기 위해 네이버 컨벤션 도입과 깃허브로 체크를 했습니다. <br> <br>- 프로젝트 진행 일정을 상황을 보며 직접 조절하고 결정하는게 어려웠고 고민하는 시간이 많았습니다.<br><br> - 회의와 문서화를 하며 프로젝트를 진행했지만 중간중간 정해진 내용이 기억이 나지 않아 햇갈리는 경우가 많았습니다.   <br><br>
- 협업 프로젝트를 해 보며 많은 걸 느낀 좋은 기회였습니다.  <br><br> - Github Flow 로 협업을 해보며 팀원들과의 소통, git 히스토리 관리의 중요성을 느겼습니다.<br> <br>- 코딩 진행 상황을 보며 코드 퀄리티나 로깅 등 디테일에서 아쉬움을 많이 느꼈습니다 <br><br> -  회의 내용을 메모하고 정리하면 편할거라 생각 했지만 정보를 이용하는게 어려웠고 문서화의 중요성을 깨달았습니다. <br><br>

- 아직 배울게 많지만 이번 프로젝트로 진행의 진행 흐름과 깃 사용, 팀장역할로 얻은 경험 등 배운게 많았던 프로젝트 였습니다.

2. 김슬기 :

- 좋았던 점<br><br>
4개월동안 웹개발을 공부하면서 배웠던 것과는 차원이 다르다고 생각했습니다.<br>
책이나 강의로 공부한것과 실제 프로젝트는 정말 많이 다르구나 느꼈고, 실제 경험해봤다는것 그 자체에 만족합니다.<br>
프로젝트를 하면서 조원들과 회의하고 협업하는 시간들 모두 즐거웠습니다. 그냥 즐기면서 했던거같네요.<br>
모난 조원 하나 없이 웃으며 끝낼 수 있는것이 가장 좋았습니다.

- 아쉬웠던 점<br><br>
잘 작동하는 코드를 작성했다고 생각했는데 오류들이 수없이 쏟아져서 심적으로 힘들었습니다...<br>
공부를 나름 했다고 해도 실제 적용해보는건 정말 다르더군요. 시행착오를 겪는 과정에서 많은걸 배웠습니다.<br>
밤새며 오류를 잡아주었던 팀원들 정말 감사합니다.

- 마지막으로<br><br>
프로젝트는 혼자 하는게 맘 편하지 않을까..? 했던 저의 생각을 바꿔주었던 시간이었습니다!<br>
어떤 팀은 싸우기도 하고 갈등도 많이 겪는다고 하던데 우리 팀이 잘 흘러갔다는건 서로서로 많은 배려를 했다는거겠네요<br>
프로젝트 하면서 짜증도 많이내고 툴툴댔는데 다 받아준 팀원들 감사합니다. 사랑합니닷..

<br><br>
3. 구현석: <br>
처음에는 지금까지 배운 내용을 제대로 활용해서 프로젝트를 진행해야한다는 것에 대해 자신감이 없었습니다. <br>
팀원 개개인의 역량 차이도 있기에 프로젝트를 시작할 때 어떻게 시작해야될 지 막막함을 느꼈습니다. <br>
그러나 역량 차이라는 것 곧 각자가 잘할 수 있는 부분이 다 다르다는 의미기도 하다는 것을 프로젝트를 진행하면서 깨닫게 되었습니다. <br>
서로 모르는 부분, 토의가 필요한 부분 등을 거리낌없이 소통하며 점점 성장하는 저를 발견할 수 있었습니다. <br>
약 한 달동안 코드를 작성하고, 수정하고, 오류를 해결하는 일련의 과정들을 팀 프로젝트라는 하나의 큰 틀 안에서 진행하는 것은 지난 몇개월 간 배우면서 연습했던 것과는 차원이 다른 경험이었습니다. <br>
끝으로 팀원분들의 허물없는 소통과 배려로 큰 갈등없이 프로젝트를 마무리할 수 있었음에 감사합니다.

<br><br>
4. 김주희: <br>
우선 팀원들이 다들 성격이 좋으셔서 프로젝트 내내 즐거웠습니다. 다들 감사합니다.<br>
- 협업<br>
협업은 처음이라 코드를 작성하기 전까지 많은 의견들을 나누고 계속해서 회의를 진행하는 부분이 가장 새로웠고,<br>
중간 테스트를 거치며 회의를 진행하는 내내 문서화를 하였음에도 누락되는 내용이 생각보다 많음을 느꼈습니다.<br><br>
- 실제 프로젝트를 경험하며<br>
책으로만 공부하던 내용들을 실제로 적용하려니 다른 부분이 너무 많아서 실제 프로젝트 내용에 따라 코드를 새롭게 작성해야 하는 부분이 처음에는 어렵게 느껴졌으나,<br>
계속해서 오류에 부딪히며 테스트하다 보니 익숙해지게 되었고 이후 오류 없이 제대로 동작할 때 큰 성취감을 느끼게 되었습니다.<br><br>
- 프로젝트의 완성 단계 <br>
완성 단계에 이르러서는 완성된 코드를 계속해서 수정하면서 어떻게 해서든 깔끔하게 작성하려 노력하였는데 이때 코드 퀄리티 부분에서 제 역량이 아직 많이 부족함을 느끼게 되었습니다.
